# CRM-система для управления продавцами и транзакциями

Проект представляет собой упрощённую CRM-систему для управления продавцами, их транзакциями и аналитикой продаж.

## Стек технологий

- Java 21
- Spring Boot
- Spring Web
- Spring Data JPA
- Hibernate
- PostgreSQL
- Gradle
- Docker Compose
- JUnit 5
- Mockito
- JaCoCo

## Основная функциональность

### Продавцы

- создание продавца;
- получение списка продавцов;
- получение продавца по id;
- обновление информации о продавце;
- логическое удаление продавца.

### Транзакции

- создание транзакции;
- получение списка всех транзакций;
- получение транзакции по id;
- получение всех транзакций конкретного продавца.

### Аналитика

- получение самого продуктивного продавца за день, месяц, квартал или год;
- получение списка продавцов, у которых сумма транзакций за выбранный период меньше указанной суммы.

## Историчность данных

В проекте используется **soft delete** для продавцов.

При удалении продавец не удаляется физически из базы данных, а помечается флагом `deleted = true`.

Это позволяет сохранить историчность данных:

- старые транзакции не теряются;
- аналитика по прошлым данным остаётся корректной;
- новые транзакции для удалённого продавца создать нельзя;
- удалённые продавцы не отображаются в обычных запросах получения продавцов.

## Запуск проекта

### 1. Клонировать проект

```bash
git clone <ссылка-на-репозиторий>
cd crm
```

### 2. Запустить PostgreSQL через Docker Compose

```bash
docker compose up -d
```

После запуска PostgreSQL будет доступен на порту, указанном в `docker-compose.yml`.

По умолчанию приложение ожидает подключение к базе:

```properties
jdbc:postgresql://localhost:5433/shift-lab-crm
```

### 3. Запустить приложение

```bash
./gradlew bootRun
```

Или через IntelliJ IDEA: запустить класс `CrmApplication`.

Приложение будет доступно по адресу:

```text
http://localhost:8080
```

## Тестовые данные

При запуске приложения база автоматически заполняется демонстрационными данными из файла:

```text
src/main/resources/data.sql
```

Скрипт очищает таблицы `sellers` и `transactions`, сбрасывает id и добавляет тестовых продавцов и транзакции.

Это сделано для удобной проверки API и аналитики.

## API

### Продавцы

#### Создать продавца

```http
POST /sellers
```

Пример тела запроса:

```json
{
  "name": "Ivan",
  "contactInfo": "ivan@mail.ru"
}
```

#### Получить всех продавцов

```http
GET /sellers
```

#### Получить продавца по id

```http
GET /sellers/{id}
```

Пример:

```http
GET /sellers/1
```

#### Обновить продавца

```http
PUT /sellers/{id}
```

Пример тела запроса:

```json
{
  "name": "Ivan Updated",
  "contactInfo": "ivan.updated@mail.ru"
}
```

#### Удалить продавца

```http
DELETE /sellers/{id}
```

Удаление выполняется логически: продавец помечается как `deleted`.

### Транзакции

#### Создать транзакцию

```http
POST /transactions
```

Пример тела запроса:

```json
{
  "sellerId": 1,
  "amount": 1500.00,
  "paymentType": "CARD"
}
```

Допустимые значения `paymentType`:

```text
CASH
CARD
TRANSFER
```

#### Получить все транзакции

```http
GET /transactions
```

#### Получить транзакцию по id

```http
GET /transactions/{id}
```

Пример:

```http
GET /transactions/1
```

#### Получить транзакции конкретного продавца

```http
GET /transactions?sellerId=1
```

### Аналитика

#### Получить самого продуктивного продавца за период

```http
GET /analytics/sellers/top-seller
```

Параметры:

| Параметр | Тип | Описание |
|---|---|---|
| `timeType` | `DAY`, `MONTH`, `QUARTER`, `YEAR` | Тип периода |
| `dateTime` | `LocalDateTime` | Дата, по которой определяется период |

Пример:

```http
GET /analytics/sellers/top-seller?timeType=MONTH&dateTime=2026-05-11T00:00:00
```

Пример ответа:

```json
{
  "sellerId": 5,
  "sellerName": "Дмитрий Соколов",
  "totalAmount": 14100.00,
  "periodStart": "2026-05-01T00:00:00",
  "periodEnd": "2026-06-01T00:00:00"
}
```

Период считается в формате `[periodStart, periodEnd)`, то есть начало включительно, конец не включительно.

#### Получить продавцов с суммой транзакций меньше указанной

```http
GET /analytics/sellers/less-than
```

Параметры:

| Параметр | Тип | Описание |
|---|---|---|
| `startPeriod` | `LocalDateTime` | Начало периода |
| `endPeriod` | `LocalDateTime` | Конец периода |
| `comparedToAmount` | `BigDecimal` | Сумма для сравнения |

Пример:

```http
GET /analytics/sellers/less-than?startPeriod=2026-05-01T00:00:00&endPeriod=2026-06-01T00:00:00&comparedToAmount=10000
```

Продавцы без транзакций за выбранный период также учитываются. Их сумма считается равной `0`.

## Обработка ошибок

В проекте реализована централизованная обработка ошибок.

Пример ответа при ошибке:

```json
{
  "errorValue": 400,
  "message": "Bad request",
  "errorDetails": [
    "name: Name must not be empty"
  ],
  "errorTime": "2026-05-12T03:32:54.11968438"
}
```

Обрабатываются основные ошибки:

- невалидные данные запроса;
- неверный тип параметра;
- неизвестный тип оплаты;
- отсутствие продавца или транзакции;
- попытка создать транзакцию для удалённого продавца.

### В проекте используются:
- Паттерн "Стратегия" для реализации выбора временного промежутка (DAY,MONTH,QUARTER,YEAR), что позволяет добавить новые вариации без большого переписывания логики.
- Централизованная обработка исключений через `@RestControllerAdvice`
- Архитектура построения "по фичам", так как по названиям всех пакетов в корне можно определить, к чему относится приложение, и с какими областями работает. В целом можно использовать и стандартную слоистую архектуру через service/controller/...
- Использовался Git Flow для создания каждого слоя в отдельной ветке feature/"название слоя"
- Использовался Hibernate, т.к. проект похож скорее на типичный CRUD, где не требуется огромной оптимизации, использовались удобства и скорость разработки с  ORM.

## Примечания

- База данных поднимается через Docker Compose.
- Таблицы создаются и обновляются Hibernate.
- Демо-данные загружаются автоматически через `data.sql`.
- Для сохранения историчности используется soft delete.
- В аналитических запросах используется группировка, агрегатные функции и фильтрация по периоду.
- При каждом запуске приложения данные будут стираться (т.к. используется truncate), сделано для удобства проверяющего.
- При получении транзакций была обнаружена проблема N+1, поэтому используются запросы с `JOIN FETCH` для её решения.
- Для ручной проверки и тестирования приветствуется использование Postman

## Что можно улучшить?

- В рамках ТЗ указано, что тип оплаты должен храниться в виде строки. При этом, я думаю, что более оптимально использовать Enum, так как у нас точно определены CASH, TRANSFER, CARD, и в том числе облегченная валидация.
- Чтобы не отходить от ТЗ, использовался String, при этом для валидации в методах сервиса используется "костыль" через:
```
Set<String> ...;
if(!Set<String>.contains(данный PaymentType));
```
- Использование `@Transactional` для реализации принципов ACID, но на данный момент в системе нет сложной работы с БД при HTTP-запросах. Так же учтена обработка запросов и исключений, поэтому оборачивать в как таковую транзакцию и делать rollback - большого смысла нет. Но в рамках масштабирования системы такая реализация допустима или даже необходима.
- Использовать миграции с помощью Flyway/Liquibase для более enterprise-подхода
- Добавить пагинацию для получения списка транзакций, продавцов, например, с помощью Pageable
- Swagger для удобства проверки
- Можно использовать Testcontainers для использования тестовых контейнеров с PostgreSQL вместо H2.
